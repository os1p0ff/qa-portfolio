## Ключевые компоненты и их назначение

(выбрано 4 компонента из схемы)

- **Клиент (браузер)** — отображает сайт и отправляет HTTP-запросы  
- **Frontend (UI)** — отвечает за пользовательский интерфейс и взаимодействие с пользователем  
- **Backend API** — обрабатывает запросы и реализует бизнес-логику  
- **Database** — хранит данные (новости, герои, обновления)

## Сравнение горячего и холодного резерва

| Критерий | Горячий резерв | Холодный резерв |
|----------|--------------|----------------|
| Готовность | Постоянно активен | Запускается при сбое |
| Синхронизация данных | В реальном времени | Периодическая |
| Время переключения | Мгновенное или минимальное | Длительное |
| Стоимость | Высокая | Низкая |
| Плюсы | Минимальный простой, высокая доступность | Экономия ресурсов |
| Минусы | Высокая стоимость реализации | Возможна потеря данных |

## Сценарий тестирования отказоустойчивости (горячий резерв)

**Компонент:** Backend API  

**Цель:**  
Проверить, что система продолжает работать при отказе основного сервера.

**Шаги:**
1. Запустить систему с основным и резервным сервером  
2. Создать нагрузку (пользователи открывают страницы)  
3. Остановить основной сервер  
4. Проверить:
   - произошло ли переключение на резервный сервер  
   - продолжается ли работа системы  
   - возникают ли ошибки  

**Ожидаемый результат:**  
Система автоматически переключается на резервный сервер, пользователи не замечают сбоя или он минимален.

---

## Метрики

- Время переключения системы  
- Количество ошибок  
- Скорость отклика системы  
- Доступность сайта
