# Задание 3. Жизненный цикл разработки ПО

## 1. Сравнение методологий разработки ПО

| Методология | Плюсы | Минусы / Риски | Пример типа проекта/компании |
| :--- | :--- | :--- | :--- |
| **Waterfall** (Каскадная модель) | • Чёткая структура и этапы<br>• Легко планировать сроки и бюджет<br>• Хорошо подходит для фиксированных требований | • Почти невозможно вносить изменения<br>• Ошибки находят поздно (на этапе тестирования)<br>• Высокий риск, если требования изменятся | Госзаказы, банковские системы. Например: разработка ПО для налоговой системы с чётким ТЗ |
| **V-Model** | • Тестирование планируется заранее<br>• Высокое качество за счёт связки этапов и тестов<br>• Подходит для QA | • Всё так же жёсткая структура<br>• Плохо переносит изменения<br>• Дорогая модель | Медицинское или авиа ПО. Например: ПО для медицинского оборудования |
| **Incremental Model** (Инкрементальная модель) | • Быстрая поставка части продукта<br>• Проще находить ошибки на ранних этапах<br>• Гибче, чем Waterfall | • Требует хорошей архитектуры<br>• Сложнее управлять зависимостями<br>• Возможны проблемы интеграции | Корпоративные системы. Например: CRM, разрабатываемая по модулям |
| **Scrum** | • Гибкость и быстрая адаптация к изменениям<br>• Частая обратная связь от заказчика<br>• Прозрачность процессов | • Требует опытной команды<br>• Сложно прогнозировать сроки<br>• Не подходит для фиксированного ТЗ | IT-стартапы, веб-сервисы. Например: мобильное приложение или онлайн-сервис |
| **Kanban** | • Непрерывная поставка задач<br>• Простая визуализация процесса<br>• Хорошо выявляет «узкие места» | • Нет жёстких сроков и ролей<br>• Может превратиться в хаос без дисциплины<br>• Сложно оценивать прогресс | Поддержка и сопровождение ПО. Например: команда поддержки продукта в IT-компании |

## 2. Agile-манифест

### Четыре основные идеи Agile-манифеста:

| Идея манифеста | Что это значит на практике? (1-2 конкретных действия команды) |
| :--- | :--- |
| **Люди и взаимодействие важнее процессов и инструментов** | 1. Ежедневные стендапы для быстрой синхронизации вместо длинных отчетов<br>2. Прямое общение разработчиков с заказчиком, минуя бюрократические цепочки |
| **Работающий продукт важнее исчерпывающей документации** | 1. Вместо 100-страничного ТЗ — короткие user stories и демонстрации рабочего прототипа<br>2. Фокус на выпуске работающих инкрементов продукта в конце каждого спринта |
| **Сотрудничество с заказчиком важнее согласования условий контракта** | 1. Регулярные встречи с продукт-оунером для уточнения приоритетов<br>2. Гибкое изменение требований в процессе работы на основе обратной связи |
| **Готовность к изменениям важнее следования первоначальному плану** | 1. Ретроспективы в конце каждого спринта для адаптации процессов<br>2. Возможность изменить бэклог спринта при появлении более важных задач |

### Комментарий о фразе: «То есть не отрицая важности того, что справа, мы всё-таки больше ценим то, что слева»

Эта фраза означает, что Agile-манифест не предлагает полностью отказаться от процессов, документации, контрактов и планов. Эти элементы остаются важными, но приоритет отдается человеческому фактору, рабочему продукту, сотрудничеству и гибкости. Например, документация нужна, но она не должна становиться самоцелью и замедлять разработку. План важен, но команда должна быть готова его корректировать, если это приведет к созданию более ценного продукта для заказчика.
