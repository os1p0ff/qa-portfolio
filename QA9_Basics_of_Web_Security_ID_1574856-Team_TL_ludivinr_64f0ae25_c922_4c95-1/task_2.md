# Task 2. Инструменты криптографии и базовая безопасность веб-приложений

## 1. Разница между кодированием, хешированием и шифрованием

### Кодирование

Кодирование — это преобразование данных в другой формат для удобства хранения или передачи информации. Оно не предназначено для защиты данных.

**Пример:** Base64 используется для передачи изображений через API.

### Хеширование

Хеширование — это преобразование данных в строку фиксированной длины. Полученный хеш невозможно обратить обратно в исходные данные.

**Пример:** хранение паролей пользователей в базе данных.

### Шифрование

Шифрование — это преобразование данных в защищённый вид с возможностью восстановления при наличии ключа.

**Пример:** HTTPS-соединение между браузером и сервером.

## 2. Симметричное и асимметричное шифрование

### Симметричное шифрование

#### Определение

Для шифрования и расшифровки используется один и тот же ключ.

#### Аналогия

Это как дверь с обычным замком: одним ключом её закрывают и этим же открывают.

#### Плюсы

- Высокая скорость работы.
- Подходит для больших объёмов данных.
- Низкие вычислительные затраты.

#### Минусы

- Нужно безопасно передать ключ.
- При компрометации ключа все данные становятся доступны злоумышленнику.

### Асимметричное шифрование

#### Определение

Используются два ключа: открытый и закрытый.

#### Аналогия

Это как почтовый ящик: любой может опустить письмо внутрь, но достать его может только владелец ключа.

#### Плюсы

- Не нужно передавать секретный ключ.
- Безопасный обмен данными через интернет.
- Возможность использования цифровой подписи.

#### Минусы

- Работает медленнее.
- Требует больше ресурсов.

## 3. Черный список алгоритмов

### Симметричное шифрование

#### DES

Уязвим к атаке полного перебора (Brute Force) из-за короткого ключа.

#### 3DES

Подвержен атаке Sweet32 из-за небольшого размера блока.

#### RC4

Уязвим к статистическим атакам на поток шифрования.

### Хеширование

#### MD5

Подвержен атакам на коллизии.

#### SHA-1

Существуют практические атаки на коллизии.

#### NTLM

Уязвим к Rainbow Table и быстрому перебору.

## 4. Белый список алгоритмов

### Симметричное шифрование

#### AES-256

Современный стандарт шифрования, устойчив к известным практическим атакам.

#### ChaCha20

Быстрый и безопасный алгоритм, широко используется в TLS.

### Хеширование

#### SHA-256

Устойчив к известным практическим коллизиям.

#### Argon2

Специально разработан для безопасного хранения паролей и защиты от перебора.

## 5. Тест-план проверки безопасности интернет-магазина

### Вход в аккаунт (логин/пароль)

#### Что проверяем

- Вход с корректными и некорректными данными.
- Пустые поля.
- SQL-инъекции.
- XSS.
- Ограничение числа попыток входа.

#### На что обращаем внимание

- Нет утечки информации о существовании пользователя.
- После множества неудачных попыток срабатывает защита.
- Пароль скрывается при вводе.

### Работа с корзиной и заказами

#### Что проверяем

- Добавление и удаление товаров.
- Изменение количества товаров.
- Попытка изменения цены через DevTools.
- Доступ к чужим заказам.

#### На что обращаем внимание

- Цена проверяется на сервере.
- Нет доступа к чужим данным.
- Итоговая сумма рассчитывается корректно.

### Оформление заказа

#### Что проверяем

- Заполнение обязательных полей.
- Некорректные данные.
- Повторную отправку заказа.
- Попытку изменения стоимости заказа.

#### На что обращаем внимание

- Не создаются дубликаты заказов.
- Данные передаются через HTTPS.
- Все проверки выполняются на сервере.

## 6. Инструменты тестировщика

### Chrome DevTools

- Анализ запросов и ответов.
- Проверка Cookies.
- Проверка Local Storage и Session Storage.

### Burp Suite

- Перехват и изменение HTTP-запросов.
- Проверка авторизации.

### Postman

- Тестирование API.
- Проверка обработки запросов.

### OWASP ZAP

- Поиск типовых уязвимостей безопасности.

## 7. Проверка ситуации: пользователь забыл выйти из аккаунта на общественном компьютере

### Как проверить

1. Войти в аккаунт.
2. Закрыть браузер без выхода.
3. Открыть браузер снова.
4. Проверить доступность аккаунта.
5. Проверить завершение сессии после периода бездействия.

### Что должно происходить для защиты

- Автоматическое завершение сессии.
- Возможность выхода со всех устройств.
- Использование Secure и HttpOnly Cookies.
- Повторный ввод пароля для критических действий.
- Ограниченное время жизни токена авторизации.
