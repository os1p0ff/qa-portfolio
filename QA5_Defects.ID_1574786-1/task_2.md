## Задание 2. Создание анти-шаблона тестирования

### Список анти-шаблонов, приводящих к пропуску критических дефектов

1. **Тестирование только "счастливого пути"** 
Проверяются только корректные сценарии использования без ошибок ввода, нестандартных действий и отказов системы. 
**Последствие:** критические баги проявляются у реальных пользователей при нестандартных действиях.

2. **Игнорирование граничных значений** 
Не проверяются минимальные, максимальные, пустые, слишком длинные или некорректные значения данных. 
**Последствие:** сбои, ошибки в расчетах, переполнения, некорректная обработка данных.

3. **Отсутствие регрессионного тестирования после изменений** 
После исправления багов или добавления новой функциональности не проверяются ранее работающие функции. 
**Последствие:** старые дефекты возвращаются, ломается стабильный функционал.

4. **Тестирование только в идеальной среде** 
Проверка проводится только на одном устройстве, браузере, ОС или при стабильном интернете. 
**Последствие:** ошибки совместимости, зависания, проблемы у пользователей на других платформах.

5. **Поверхностная проверка интеграций** 
Недостаточно тестируется обмен данными между сервисами, API, базами данных, платежными системами. 
**Последствие:** потеря данных, некорректные статусы заказов, ошибки синхронизации.

6. **Отсутствие проверки под нагрузкой** 
Система тестируется только при небольшом количестве пользователей и запросов. 
**Последствие:** падения сервиса, медленная работа, недоступность в пиковые часы.

7. **Нечеткие требования и отсутствие уточнений** 
Тестирование начинается без понимания бизнес-логики и ожидаемого результата. 
**Последствие:** пропуск логических дефектов и спорные баг-репорты.
