## Задание 4. Жизненный цикл баг-репорта

## Таблица этапов жизненного цикла баг-репорта

| Этап / статус | Что происходит | Кто и на каком этапе может менять статус | Пример |
|---------------|----------------|------------------------------------------|--------|
| New | Дефект создан, но еще не взят в работу | Автоматически или QA при создании бага | QA создал новый баг в Jira |
| Open | Баг подтвержден и передан в работу | QA Lead / Manager / система workflow | После триажа баг отправлен разработчику |
| In Progress | Разработчик начал исправление | Developer | Разработчик взял задачу в работу |
| Fixed | Исправление внесено, задача готова к проверке | Developer | Код исправлен и отправлен на тест |
| Ready for QA | Исправление выкачено на тестовую среду | Developer / DevOps | Новый билд доступен QA |
| Retest | Тестировщик повторно проверяет исправление | QA | QA запускает повторную проверку |
| Verified | Баг исправлен и подтвержден тестировщиком | QA | Ошибка больше не воспроизводится |
| Reopened | Баг снова воспроизводится после исправления | QA | После проверки проблема осталась |
| Closed | Работа по дефекту завершена | QA Lead / Manager / QA | Исправленный баг закрыт |
| Duplicate | Такой баг уже существует | QA / Lead | Найден ранее зарегистрированный дефект |
| Not a Bug | Поведение признано корректным | Developer + QA + PO | Ожидаемое поведение системы |
| Cannot Reproduce | Ошибка не воспроизводится | Developer / QA после проверки | На тестовой среде ошибка не найдена |
| Deferred | Исправление перенесено на будущий релиз | Manager / Product Owner | Баг не критичен для текущего релиза |
| Won’t Fix | Принято решение не исправлять | Product Owner / Manager | Исправление слишком дорогое при низком риске |
| Blocked | Исправление временно невозможно | Developer / Manager | Зависимость от стороннего сервиса |

## Конфликтные ценарии

## Сценарий 1: Разработчик говорит "Not a bug (Не ошибка/Не дефект)"

### Стратегия действий

1. Спокойно уточнить ожидаемую бизнес-логику.
2. Показать шаги воспроизведения.
3. Приложить скриншот / видео / логи.
4. Сравнить с требованиями или макетом.
5. При необходимости подключить аналитика или PO.

### Пример коммуникации

> Привет. По шагам 1-2-3 проблема стабильно воспроизводится. 
> Согласно ожидаемому сценарию кнопка должна быть активна после заполнения поля. 
> Можем свериться с требованиями или подключить PO для уточнения?

---

## Сценарий 2: Баг не воспроизводится у разработчика

### Стратегия действий

1. Уточнить окружение разработчика (браузер, версия ОС, билд).
2. Проверить различия среды.
3. Записать видео воспроизведения.
4. Передать точные тестовые данные.
5. Попробовать совместный созвон и live reproduction.

### Пример коммуникации

> У меня дефект воспроизводится на Chrome 123 / Android 14 / build 1.5.2. 
> Подскажи, пожалуйста, какое у тебя окружение? 
> Прикладываю видео и тестовый аккаунт для проверки.

## Использование ИИ для ролевого сценария

### Промпт

Сыграй роль разработчика, который считает "Not a bug".
Контекст:
На странице регистрации после заполнения всех обязательных полей кнопка Submit остается неактивной. Отвечай как реальный разработчик.
Не задавай уточняющих вопросов.
Не предлагай очистить кеш, инкогнито, переустановку браузера и другие стандартные советы. Спорь короткими сообщениями
