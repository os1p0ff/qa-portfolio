# Задание 2. Уровни тестирования

| Уровень тестирования | Что проверяем и на каком этапе | Кем проводится | Типичные артефакты для тестирования | Пример дефекта |
|---|---|---|---|---|
| **Модульное (Unit)** | Проверяется работа отдельных функций и методов на этапе разработки. | Разработчики | исходный код, техническое задание, документация API | 1. Неверная формула расчета комиссии. <br>2. Функция перевода возвращает ошибку при нулевой сумме. <br>3. Некорректная обработка исключения при отсутствии счета. |
| **Интеграционное** | Проверяется взаимодействие модулей между собой после их объединения. | Разработчики и тестировщики | API-документация, архитектура системы, тестовые данные | 1. Модуль перевода не получает данные о балансе. <br>2. Ошибка передачи данных между сервисом авторизации и сервисом платежей. <br>3. Дублирование операции при повторном запросе. |
| **Системное** | Проверяется вся система полностью на тестовой среде перед релизом. | QA-команда | требования, тест-кейсы, чек-листы, тестовые сценарии | 1. Приложение вылетает при переводе крупной суммы. <br>2. Интерфейс отображается некорректно на маленьком экране. <br>3. Медленная загрузка списка счетов. |
| **Приемочное (Acceptance)** | Проверка продукта перед передачей заказчику или выпуском для пользователей. | Заказчик, бизнес-аналитики, QA | бизнес-требования, пользовательские сценарии, acceptance-критерии | 1. Пользователь не может найти кнопку перевода. <br>2. Приложение не соответствует требованиям безопасности. <br>3. Отсутствует нужный тип перевода между счетами. |
